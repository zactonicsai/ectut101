▸ What’s actually happening?

Primary shards split your index across the cluster. With 2 shards, half your documents go to one node, half to the other. More shards = more parallelism but more overhead.

Replicas are copies on other nodes. Setting replicas=1 on a 2-shard index creates 4 total shards (2 primary + 2 replica). If one node dies, the other still has every document.

text vs keyword: text fields get tokenized and lowercased (search "machines" and find "machine"). keyword fields are stored literally — perfect for tags, IDs, and filtering.

Simulate node failover

Step-by-step drill

  1. 1
    Open Tab ④ Cluster & Shards in another window. Note every shard is STARTED and assigned to a node.
  2. 2
    In your terminal, kill one Elasticsearch node: 
    docker stop es02
  3. 3
    Within ~10s the cluster health bar turns yellow — replicas for shards that were on es02 are now UNASSIGNED. Data is still fully available; reads come from the surviving primaries.
  4. 4
    Check the watchdog logs to see the failover recorded: 
    docker logs -f watchdog
  5. 5
    Run a search here or via the Search tab — it keeps working. The Python client automatically uses the healthy node.
  6. 6
    Bring it back: 
    docker start es02
    Watch the shards return to green over ~30s as replicas are rebuilt.

Why this works

When you created the index with replicas: 1, the cluster allocated:

shard-0 primary  → es01   shard-0 replica  → es02
shard-1 primary  → es02   shard-1 replica  → es01

When es02 disappears, the cluster already has a full copy of shard-1 on es01 — it promotes that replica to primary instantly. Writes keep working, reads keep working.

This is why replicas >= 1 is the minimum for production. With replicas=0, a single node dying means permanent data loss on whatever shards it held.

The Go watchdog is running in its own container and polls both nodes every 3s — in a real system that signal would page on-call or trigger reallocation scripts.
