CHAPTER 04

The ingestion pipeline

One-time work, run for every document you want to make searchable. Five steps. Each one has a reason.

1Load — extract the text

Get raw text out of whatever source you have. The gotcha is that different formats need different extractors, and some preserve structure (headings, tables) while others flatten everything.

SourceToolPreserves structure?
PDF (text)pypdf, pdfplumber, pymupdfPartial
PDF (scanned)tesseract OCR, AWS TextractNo
HTMLBeautifulSoup, trafilaturaYes (headings)
DOCX / PPTXpython-docx, python-pptxYes
Markdownmarkdown-it, nativeYes
Confluence / Notionofficial REST APIYes
DatabasesSELECT query + templateDepends on schema
PDF PAIN

PDFs are the hardest input. Two-column layouts get read as one long stream with interleaved columns. Tables get flattened. Scanned PDFs (image-based) need OCR, which is slow and error-prone. If your project is PDF-heavy, budget significant time for this step — it’s where quality is won or lost.

2Clean — strip the noise

Raw extracted text is full of noise: HTML nav menus, PDF page headers/footers, “Click here” boilerplate, table-of-contents leakage. Clean it before embedding — otherwise the model wastes capacity encoding junk that drowns out the real signal.

01# Rough cleaning recipe
02text = text.strip()
03text = re.sub(r'\n{3,}', '\n\n', text)     # collapse 3+ newlines
04text = re.sub(r'Page \d+ of \d+', '', text) # drop footers
05text = re.sub(r' ', ' ', text)
06text = ftfy.fix_text(text)                     # fixes mojibake
07text = unicodedata.normalize('NFKC', text)    # unify quotes, dashes

3Chunk — split into retrievable pieces

The most consequential step. A chunk is the unit that gets embedded, stored, retrieved, and handed to the LLM. Chunks that are too big lose focus. Chunks too small lose context. Get this wrong and the whole system suffers.

3.1   Why can’t we just embed the whole document?

Reason 1 · Token limits

Embedding models have a max input length (256, 512, 8192 tokens). A 50-page PDF doesn’t fit. You’d lose everything past the limit.

Reason 2 · Semantic dilution

A long document covers many topics. Averaging everything into one vector gives a blurry “average meaning” that matches no specific question well.

Reason 3 · Context window

Even if you retrieved it, you couldn’t fit a 50-page document into the LLM’s prompt without using up your whole budget.

Reason 4 · Citation granularity

If you retrieve “the whole doc,” you can’t point a user at the specific passage. Chunks enable fine-grained citations.

3.2   Chunking strategies

StrategyHowQualityWhen to use
Fixed sizeEvery N charactersPoorNever, except proof of concept
Sentence splitBreak on . ! ?OKQ&A style content
Recursive characterSplit on \n\n, then \n, then . , wordGoodDefault for most content
Markdown-awareSplit on ## and ### headingsGreatStructured docs, wikis
SemanticEmbed sentences, cluster by similarityGreatResearch papers, prose
Agentic / LLMAsk an LLM where to splitBestCritical apps, complex docs

3.3   Chunking with overlap — the picture

RULES OF THUMB
  • Chunk size: 300–800 tokens for most text. Shorter = more precise retrieval but less context per chunk.
  • Overlap: 10–20% of the chunk size. Prevents sentences being split at the worst moment.
  • Respect boundaries: prefer splitting on paragraph or sentence ends, not mid-word.
  • Include context: prepend the document title / section heading to every chunk. A chunk that says “See the algorithm above” is useless — add the heading so it reads “Chapter 5: HNSW — See the algorithm above.”

4Embed — call the model

Hand each chunk to the embedding model and store the returned vector. Two practical concerns: batching and caching.

01from sentence_transformers import SentenceTransformer
02model = SentenceTransformer("BAAI/bge-base-en-v1.5")
03
04# Batch up to 32-64 chunks per call — 10-100x faster than one-at-a-time
05vectors = model.encode(
06    chunks,
07    batch_size=32,
08    normalize_embeddings=True,   # unit-length → cosine == dot product
09    show_progress_bar=True,
10)
11# vectors is a numpy array of shape (N, 768)
WHY batch

GPUs love parallelism. Encoding 32 chunks in one forward pass is almost as fast as encoding one — the matrix ops fill the chip. Sending them one at a time wastes 90%+ of GPU throughput. For hosted APIs, batching also reduces network round-trip overhead.

CACHE AGGRESSIVELY

Embedding is the slowest and usually most expensive part of ingestion. Store a hash of each chunk’s text. If you see that hash again (re-ingest, retry, partial update), reuse the cached vector. Saves money on hosted APIs and makes pipeline re-runs fast.

5Index — write to Elasticsearch

Every chunk becomes one document in Elasticsearch, containing the original text, the vector, and any metadata you want to filter by later.

01from elasticsearch.helpers import bulk
02
03actions = [
04    {
05        "_index": "knowledge",
06        "_id": chunk_id(doc_id, i),
07        "_source": {
08            "doc_id":    doc_id,
09            "chunk_idx": i,
10            "title":     metadata["title"],
11            "section":   metadata["section"],
12            "text":      chunk_text,             # the raw text
13            "embedding": vector.tolist(),        # 768 floats
14            "source":    "elastic-internals.md",
15            "updated":   "2026-04-23",
16        },
17    }
18    for i, (chunk_text, vector) in enumerate(zip(chunks, vectors))
19]
20bulk(es, actions, chunk_size=500)
KEY MOVES
  • Deterministic IDs. Use chunk_id = hash(doc_id, chunk_idx). Re-ingesting the same doc overwrites the existing chunks — no duplicates.
  • Store the text. The vector alone is useless for returning to the user. Always store the chunk text alongside.
  • Store metadata. doc_id, source URL, date, author, category — anything you might filter by later. Cheap now, priceless at query time.
  • Use the bulk API. 500 docs per call is a good batch size. Single-doc indexing is 50× slower.
CHAPTER 01

The problem — why an LLM alone isn't enough.

Before we build anything, we need to understand precisely what problem RAG solves. A large language model on its own is remarkable at reasoning, writing, and summarizing — but it has three hard failure modes when you ask it about your data.

PROBLEM 01

The cutoff.

An LLM was trained on data frozen at a specific date. If you ask about your company's Q3 2026 earnings, a product launched last week, or a private Slack thread from this morning — it cannot know. Its knowledge is a snapshot.

PROBLEM 02

The hallucination.

When an LLM doesn't know something, it doesn't say so. It generates plausible-sounding text because that's what it was trained to do. Ask it about your internal API and it will confidently invent endpoint names, parameters, and return values.

PROBLEM 03

The window.

Every model has a context window — a maximum amount of text (tokens) you can feed it at once. Even a 200,000-token window can't hold a 10-million-word codebase or every document in your company wiki. You can't just "paste it all in."

The core idea of RAG

Instead of trying to cram everything into the model, we retrieve only the most relevant few paragraphs for each question and paste those into the prompt. The LLM then generates its answer grounded in the snippets we provided — not in whatever it half-remembers from training.

This turns the LLM from a memorization engine into a reasoning engine over fresh data. It's the single most important pattern in practical LLM engineering.

DEFINITION · "GROUNDING"

When we say an answer is grounded, we mean every factual claim in it can be traced back to a specific retrieved passage. The opposite is ungrounded output, where the model makes a claim whose origin is "somewhere in my training data" — which is often indistinguishable from "nowhere, I made it up."

CHAPTER 02

What an embedding actually is.

Everything in RAG rests on one idea: we can turn a paragraph of text into a fixed-length list of numbers such that paragraphs with similar meanings have similar lists. That list is called an embedding or a vector. The list is typically between 384 and 3,072 numbers long.

Before we look at formulas, let's build intuition with a simple visualization. Imagine every document you own represented as a single dot in space. The position of each dot is chosen so that meaning equals proximity.

What a real embedding looks like.

Let's make this concrete. Here is the output when we embed two sentences using OpenAI's text-embedding-3-small model (1,536 dimensions): 

# Input sentence 1
text = "A golden retriever plays fetch in the park."

# After calling the embedding API:
vector = [
  -0.0142,  0.0387, -0.0259,  0.0091, -0.0503,
   0.0224, -0.0118,  0.0647, -0.0312,  0.0449,
   # ... 1,526 more numbers ...
   0.0083, -0.0195,  0.0372, -0.0054,  0.0218,
]
# len(vector) == 1536

# Input sentence 2 (similar meaning)
text2 = "The lab ran after a tennis ball on the grass."
# → a DIFFERENT list of 1,536 numbers, but mathematically CLOSE to the first
BEGINNER QUESTION

"But what do those specific numbers mean? Is the 47th number how 'doggy' the sentence is?"

No. The individual dimensions are not interpretable. The model learned, during training on billions of sentences, to arrange them so that the overall geometric position encodes meaning. Think of it like latitude-and-longitude: the number 40.7 doesn't mean anything on its own, but (40.7, -74.0) together pinpoints New York. Embeddings are the same idea, except with 1,500 dimensions instead of 2.

Key properties of embeddings.

PropertyWhat it means
Fixed lengthA 100-word paragraph and a 3-word phrase both produce a vector of the same dimension (e.g., 1,536). This is what makes them comparable.
DenseAlmost every number is non-zero. (Contrast with "sparse" representations like TF-IDF, which are mostly zeros.) Dense vectors pack meaning into every dimension.
ContinuousEach number is a real-valued float, not a category. Small changes in meaning produce small changes in position.
LearnedThe model wasn't programmed with rules about meaning. It learned the geometry by being trained on enormous amounts of text with an objective that encourages similar-meaning inputs to produce similar outputs.
Model-specificVectors from different models are not comparable. An embedding from OpenAI cannot be compared to one from Cohere. You must use the same model for both your indexed documents and your queries.

The retrieval problem — "find documents similar to this question" — has now become a geometric problem: "find the points in N-dimensional space closest to this query point." That's a problem we know how to solve efficiently, and it's what the rest of this tutorial builds.

CHAPTER 05

HNSW — the algorithm that makes vector search fast

You now have millions of 768-dimensional vectors in Elasticsearch. A user sends a query vector. How do you find the top-10 closest vectors without comparing against all million? Enter HNSW.

5.1   The naive approach and why it fails

The obvious answer: compute the similarity of the query to every stored vector, then sort and take the top 10. This is called exact k-nearest-neighbors or brute force search.

COMPLEXITY

For N vectors of dimension d, brute force is O(N · d) per query. With 10M vectors of 768 dims, that’s 7.7 billion multiply-adds per query — hundreds of milliseconds even on a beefy server. Unusable for interactive apps.

We need an approximate nearest-neighbor algorithm: one that finds the top-K probably correct in O(log N) time, trading a tiny amount of accuracy for a massive speedup. The dominant algorithm today is HNSW — Hierarchical Navigable Small World graphs.

5.2   The HNSW idea in one picture

Imagine a highway system. Major interstates connect distant cities quickly. Local roads give fine-grained access within a city. To travel from one house to another far away, you don’t drive through every street — you get on the interstate, cover most of the distance fast, then exit and drive locally.

HNSW builds a similar structure over the vectors. A top layer has a few vectors with long-range connections. Lower layers have more vectors with shorter-range connections. The bottom layer has every vector. Search starts at the top, greedily walks toward the query, descends a layer, and repeats — zooming in until it finds the true neighborhood.

5.3   Key parameters

ParameterWhat it controlsDefaultBigger = ?
mMax connections per node in each graph layer16Better recall, more RAM, slower build
ef_constructionCandidate list size during build100Better graph quality, slower ingestion
ef_searchCandidate list size during query (a.k.a. num_candidates in ES)variesBetter recall, slower query
kNumber of final results to return10More results, small cost
TUNING RULE

Start with m=16, ef_construction=100, num_candidates=10·k (so for k=10, use 100). If recall is too low, increase num_candidates. If ingestion is too slow, lower ef_construction. Only touch m if you’ve understood the memory impact — higher m multiplies the graph’s footprint.

COMPLEXITY

HNSW query time is O(log N) on average — compare that to O(N) for brute force. On a 10M-vector index, HNSW takes ∼5 ms, brute force takes ∼500 ms. That’s the 100× speedup that makes real-time RAG possible. The tradeoff is recall — typically 95-99% of the true top-K is returned, which is almost always fine for RAG.

CHAPTER 06

Elasticsearch as the vector database

Elasticsearch has had full-text search since 2010. Since version 8.0 it also has native dense vectors and HNSW-backed kNN — meaning one system holds both your keyword index and your vector index.

6.1   Defining a vector-enabled index

The dense_vector field type tells Elasticsearch: this field contains a fixed-dimension float array, build an HNSW graph over it, enable kNN queries.

08      "title":     { "type": "text" },
09      "section":   { "type": "keyword" },
10      "text":      { "type": "text", "analyzer": "english" },
11      "source":    { "type": "keyword" },
12      "updated":   { "type": "date" },
13      "embedding": {
14        "type": "dense_vector",
15        "dims": 768,

17        "similarity": "cosine",

LINE-BY-LINE

6.2   Running a kNN query

Once documents are indexed, a kNN query looks like this:

01GET /knowledge/_search
02{
03  "knn": {
04    "field": "embedding",
05    "query_vector": [0.0234, -0.1812, 0.0457, ...],   # 768 floats
06    "k": 10,
07    "num_candidates": 100
08  },
09  "_source": ["text", "title", "source", "doc_id"]
10}
THE TWO NUMBERS YOU CHOOSE

k is how many results you want back. num_candidates is how hard HNSW searches before returning. ES collects num_candidates near neighbors per shard, sorts them, and returns the top k. Higher num_candidates = better recall but slower query. Rule of thumb: num_candidates = 10 × k.

6.3   Filtering + kNN (pre-filter)

Real queries combine semantic search with structured filters: “find chunks similar to the question, but only from docs updated in the last 30 days.” Elasticsearch supports this natively.

01GET /knowledge/_search
02{
03  "knn": {
04    "field": "embedding",
05    "query_vector": [...],
06    "k": 10,
07    "num_candidates": 100,
08    "filter": {
09      "bool": {
10        "must": [
11          { "range": { "updated": { "gte": "2025-04-01" } } },
12          { "term":  { "section": "elasticsearch-internals" } }
13        ]
14      }
15    }
16  }
17}
WHY this matters

ES runs the filter inside the kNN walk — it only considers matching documents when traversing the HNSW graph. This is much more efficient than retrieving top-K then filtering (which might return zero matches). Use filters liberally: date ranges, tenant IDs, document categories, access control.

CHAPTER 03

The math of similarity.

"Find the closest points" requires a definition of "close." There are three candidates in common use; two of them are actually the same thing when vectors are normalized. Let's work through each.

3.1   Euclidean distance — the ruler

The familiar straight-line distance from high school geometry. For two vectors a and b with n components:

d(a, b) = √( (a1 − b1)² + (a2 − b2)² + ... + (an − bn)² )

Smaller number = closer. Works fine, but has one problem for text: it is sensitive to magnitude. A very long document might have a vector with larger values overall, making it artificially "far" from a short query even when they mean the same thing. That's why we usually use cosine similarity instead.

3.2   Cosine similarity — the angle

Instead of asking "how far apart are the endpoints," cosine similarity asks "how similar are the directions the vectors point?" Mathematically, it's the cosine of the angle between them.

cos(θ) = ( a · b ) / ( ‖a‖ × ‖b‖ )

where
  a · b = a1b1 + a2b2 + ... + anbn ← dot product
  ‖a‖ = √( a1² + a2² + ... + an² ) ← length (magnitude)

3.3   Dot product — the fast path

The dot product is just the numerator of cosine similarity:

a · b = a1b1 + a2b2 + ... + anbn

Computing it is cheaper than cosine because you skip the two square-roots and a division. When vectors are pre-normalized — meaning their length has been scaled to 1.0 ahead of time — the dot product is mathematically equivalent to cosine similarity but ~3× faster. Almost every modern embedding model returns normalized vectors, so you get the best of both.

IN ELASTICSEARCH

Elasticsearch lets you pick at index-creation time: "similarity": "cosine", "dot_product", or "l2_norm" (Euclidean). For most text RAG applications with a modern embedding model: use cosine (simple and correct) or dot_product (faster, but you must normalize).

3.4   A worked example

Let's pretend embeddings are just 3-dimensional to make the arithmetic tractable. In reality they're 1,000+ dimensions but the operation is identical. 

# Three fake "embeddings" in 3-D
query      = [0.6, 0.8, 0.1]  # "dog running"
candidate1 = [0.7, 0.7, 0.2]  # "puppy playing"      - likely similar
candidate2 = [-0.5, 0.3, 0.9] # "tax accounting"     - likely different

# Dot product (candidate 1)
dot1 = (0.6 × 0.7) + (0.8 × 0.7) + (0.1 × 0.2)
     = 0.42 + 0.56 + 0.02
     = 1.00

# Dot product (candidate 2)
dot2 = (0.6 × -0.5) + (0.8 × 0.3) + (0.1 × 0.9)
     = -0.30 + 0.24 + 0.09
     = 0.03

# Conclusion: candidate1 (score 1.00) is vastly closer to the query than candidate2 (score 0.03).
# We rank by this score and return the top K.
WHY THE MATH MATTERS

Every vector database in the world — Elasticsearch, Pinecone, Weaviate, Qdrant, Milvus, pgvector — is at its core a data structure that answers the question "which of my millions of stored vectors have the highest dot product with this query vector?" as fast as possible. Everything else is engineering around that core.

CHAPTER 07

Hybrid search — BM25 meets vectors

Vector search is brilliant at meaning. Keyword search is brilliant at exact terms (product IDs, error codes, proper names). Hybrid search runs both and fuses their results. Almost always beats either one alone.

7.1   BM25 in one paragraph

BM25 (“Best Match 25”) is the keyword-scoring function Elasticsearch has used forever. Given a query and a document, it computes a score that grows with (a) how many query terms appear, (b) how rare those terms are across the index, and (c) how appropriate the document length is.

THE FORMULA
score(q, d) = ∑t ∈ q IDF(t) · tf(t, d) · (k1 + 1)tf(t, d) + k1 · (1 − b + b · |d| / avgdl)

Where tf(t,d) is term frequency, IDF(t) = log((N − df + 0.5)/(df + 0.5) + 1) is inverse document frequency, |d| is doc length, avgdl is average doc length, and k1=1.2, b=0.75 by default. You don’t need to memorize this — ES computes it automatically when you do a match query.

7.2   When vectors miss and BM25 saves you

Vectors win when…

Query: “how do I make my cluster bigger?” — none of those words appear in the right doc (“add data nodes to distribute shards”). Vector captures intent.

BM25 wins when…

Query: “error E_SHARD_UNASSIGNED_5” — an exact code. Embedding models often smear these; BM25 finds the literal string instantly.

7.3   Reciprocal Rank Fusion (RRF)

So you ran BM25 and got a ranked list. You also ran kNN and got another ranked list. How do you combine them? You could try to add the raw scores, but they’re on different scales (BM25 might go from 2 to 15, cosine from 0.4 to 0.95). Normalizing is brittle.

Reciprocal Rank Fusion sidesteps the problem by ignoring scores entirely and using only ranks. For each document, sum 1 / (k + rank) across the lists it appears in:

RRF_score(d)  =  ∑r ∈ rankers   1k + rankr(d)

k is a constant, typically 60. It damps the contribution of low-ranked results. Sort documents by total RRF score descending; that’s your fused ranking.

Worked example (k = 60)

Doc BM25 rank 1/(60+rBM25) Vector rank 1/(60+rvec) RRF sum Final
doc_A10.016430.01590.0323#1
doc_B20.016150.01540.0315#2
doc_C—010.01640.0164#3
doc_D40.0156—00.0156#4
WHY it works so well

Documents that appear in both lists get a big boost (evidence from both signals). Documents in only one list still appear but are ranked lower. The algorithm has no parameters to tune (k=60 is industry standard), requires no score normalization, and consistently outperforms more elaborate schemes.

7.4   Hybrid query in Elasticsearch

Elasticsearch has RRF built in via the rrf retriever (since 8.14):

01GET /knowledge/_search
02{
03  "retriever": {
04    "rrf": {
05      "retrievers": [
06        { "standard": {                        # BM25 lane
07            "query": { "match": { "text": "shard allocation" } } } },
08        { "knn": {                             # vector lane
09            "field": "embedding",
10            "query_vector": [...],
11            "k": 50, "num_candidates": 200 } }
12      ],
13      "rank_window_size": 50,
14      "rank_constant": 60
15    }
16  }
17}
WHAT happens

ES runs both inner retrievers in parallel, takes the top rank_window_size from each, computes RRF with rank_constant = 60, and returns a single ranked list. You get the best of both search paradigms with one query.

CHAPTER 08

Query-time flow — from question to context

This is what happens every time a user asks something. Seven sub-steps, typically completing in 200–800 ms.

8.1   Embed the query

Use the exact same embedding model and preprocessing as you did at ingest time. Different model = vectors in a different space = garbage similarity scores.

01query = "How does shard allocation work?"
02query_vec = model.encode(query, normalize_embeddings=True)
03# Some models want you to prefix queries, e.g. bge-base:
04# query = "Represent this question for searching: " + query
05# Check the model card — getting this wrong degrades quality ~5-10%.

8.2   Retrieve (hybrid via RRF)

Send the query vector and the raw query text to Elasticsearch in one hybrid request (Chapter 7). Retrieve the top 50 chunks, not just 5 — you want oversupply so the reranker has material to work with.

8.3   Rerank (optional but powerful)

Bi-encoder embeddings (the kind we’ve been using) are fast because they encode query and document independently. The cost: they’re less accurate. A cross-encoder takes the query and a candidate chunk together, runs them through a transformer, and outputs a relevance score. Much slower but significantly more accurate.

THE TWO-STAGE STRATEGY

Use the fast vector search to retrieve, say, the top 50 candidates. Run a cross-encoder (e.g. BAAI/bge-reranker-base, or Cohere Rerank v3) on those 50 to rescore them. Keep the top 5. You get near-cross-encoder accuracy without paying its cost at scale.

8.4   Why retrieve 50 then narrow to 5?

Three compounding reasons. First, the retriever can miss the perfect chunk in position 1 but catch it in position 18 — reranking rescues it. Second, you want diverse candidates so the LLM sees evidence from multiple sources. Third, the LLM has a context window — you can’t stuff 50 chunks, but you can stuff 5. Reranking is the filter that picks the best 5.

CHAPTER 09

Prompt construction — stuffing the context

The retriever found your top chunks. Now you have to present them to the LLM so it produces a good answer grounded in that evidence and not hallucinating alongside it.

9.1   The anatomy of a RAG prompt

A well-formed RAG prompt has four sections in this order:

01# ============ SYSTEM MESSAGE (instructions to the model) =============
02You are a helpful technical assistant. Answer the user’s question
03using ONLY the provided context passages. If the answer is not in
04the context, say “I don’t know based on the provided docs.”
05Cite sources by their [id] at the end of each claim.
06
07# ============ CONTEXT (retrieved chunks, numbered) ===================
08[1] (from elastic-internals.md, section “Shard Lifecycle”)
09The master node assigns shards to data nodes using the

9.2   Design rules

Rule 1 · Be explicit about grounding

The phrase “using ONLY the provided context” does serious work. Without it, the model blends memory with context and you get silent hallucinations.

Rule 2 · Number the chunks

Numbered [1] [2] [3] tags give the model a clean handle for citing sources. Ask it to cite — and it will.

Rule 3 · Allow “I don’t know”

Give the model permission to abstain. Without this, it will invent an answer rather than appear unhelpful. This single line prevents most hallucinations.

Rule 4 · Temperature = 0

For factual RAG, set the sampling temperature low (0 or 0.1). Creativity is the enemy here — you want the model to stay close to the evidence.

Rule 5 · Include section / title context

Prefix each chunk with its doc name and section heading. Helps the model disambiguate when the same term means different things in different docs.

Rule 6 · Put the question LAST

Models attend most strongly to the beginning and end of a prompt (“primacy” and “recency” effects). Context in the middle, question at the end.

9.3   Context window math

Every model has a maximum context length in tokens. Your prompt must fit within it, leaving room for the output.

available_tokens  =  context_window − system_tokens − question_tokens − output_reserve
Example with Claude Sonnet 4.5 (200k context)
context_window    = 200,000
system_tokens     =     200
question_tokens   =      30
output_reserve    =   4,000   # how much room we leave for the answer
available for context = 195,770 tokens
# ~150 chunks of 800 tokens each. Plenty.
“LOST IN THE MIDDLE”

Research shows LLMs attend more strongly to the start and end of a long context than the middle. If you’re stuffing 100+ chunks, put the most important ones first and last. Or — better — don’t stuff 100 chunks. Use reranking to whittle to the best 5-10 and give them full attention.

MORE IS NOT BETTER

Resist the urge to stuff the whole context window. Every extra irrelevant chunk is noise that drags down answer quality. 3-8 focused chunks usually beats 50 loosely-relevant ones. Quality > quantity.

Final answer returned to user

Shard allocation in Elasticsearch is handled by the AllocationService on the master node [1]. The process works in two main phases:

Initial placement. Primary shards are placed first, then replicas are placed on different nodes to ensure redundancy. The allocator weighs factors including disk usage, current shard count per node, and configured awareness attributes [2].

Rebalancing. When a node joins or leaves the cluster, the allocator re-plans shard placement and issues move commands to redistribute shards [3]. These moves are throttled by cluster.routing.allocation.node_concurrent_recoveries, which defaults to 2 [5].

Operators can override automatic decisions by issuing explicit move, cancel, or allocate commands via the _cluster/reroute API [4].

What to notice
  • Every claim has a citation. Users can audit the answer against the source docs.
  • No hallucinations. The model didn’t invent mechanisms it hadn’t seen (no “BALANCED_PLUS” fiction from Chapter 1).
  • Structure emerged naturally. The model organized the context into “initial placement” and “rebalancing” because the retrieved chunks pointed at both phases.
  • Tangential detail included (reroute API) because it was retrieved and the model considered it relevant.
CHAPTER 11

Pitfalls & tuning — what breaks, how to fix it

Every RAG system has the same failure modes. Knowing what to look for saves weeks of debugging.

11.1   The six failure modes

1
“It returns irrelevant chunks.”

Chunk size is wrong, or the embedding model doesn’t know your domain. First, check that chunks are 300–800 tokens and cut at semantic boundaries. Second, check the MTEB leaderboard for models in your language/domain. Third, add a reranker — it fixes most relevance issues cheaply.

2
“It finds nothing for specific IDs or codes.”

Pure vector search smears exact tokens. Add a BM25 lane via hybrid retrieval + RRF (Chapter 7). Literal queries like error codes, product SKUs, and filenames need keyword matching.

3
“The LLM still hallucinates.”

Two causes. (a) Retrieved chunks don’t actually contain the answer — fix retrieval first. (b) Your prompt doesn’t constrain grounding. Add explicit instructions: “Use ONLY the context.” “If the answer is not in the context, say so.” Set temperature to 0. Ask for citations.

4
“It returns stale answers after I update the docs.”

You forgot to re-ingest. Set up a pipeline (cron, webhook, or change-data-capture) that catches source updates and re-embeds changed chunks. Use deterministic chunk IDs so updates overwrite instead of duplicating.

5
“Duplicate or near-duplicate results crowd out diversity.”

Happens with overlapping chunks or near-identical documents. Use MMR (Maximal Marginal Relevance) to diversify: pick the first result greedily, then for each subsequent pick, penalize similarity to already-selected results. Or deduplicate by doc_id at the retrieval layer.

6
“Latency is too high.”

Profile first. The LLM generation is almost always dominant. Shorten the prompt (fewer chunks, tighter system message). Stream tokens to the client so time-to-first-token is the felt latency. On the retrieval side, lower num_candidates and skip rerank if the raw ranking is good enough.

11.2   Evaluation — how do you know it works?

The uncomfortable truth: without evaluation, you’re guessing. Build a test set of ~50–100 real questions with known good answers and sources. Then measure three things:

Retrieval metrics

Recall@k: does at least one correct chunk appear in the top k?
MRR: mean reciprocal rank of the first correct chunk.

Groundedness

Does every claim in the answer have supporting text in the retrieved context? Measured with LLM-as-judge or tools like RAGAS.

Answer correctness

Compared to the gold-standard answer, does the produced answer cover the key facts? Also LLM-judged or human-judged.

Iterate, don’t guess

Every change (new chunking strategy, new model, added reranker, new prompt) should be measured against the eval set. If the metric doesn’t move or regresses, don’t ship. This discipline separates working RAG systems from vibes-based RAG systems.

11.3   The tuning checklist

  1. 01Start with bge-base-en-v1.5, 768 dims, cosine, chunks 500 tokens with 100 overlap. That’s a strong baseline.
  2. 02Build a 50-question eval set BEFORE tuning. Without it you’re flying blind.
  3. 03Add hybrid (BM25 + kNN via RRF). Almost always beats either alone.
  4. 04Add a reranker on top-50. Biggest accuracy gain for lowest engineering cost.
  5. 05Tune chunk size in ranges [256, 512, 1024]. Measure on your eval set.
  6. 06Tighten prompt: “only from context,” “say I don’t know,” numbered citations, temperature 0.
  7. 07Add filters (date, tenant, category) if you have structured metadata.
  8. 08Only after all the above: try larger / hosted embedding models. Diminishing returns.
  9. 09Stream LLM output. Perceived latency often dominates real latency.
  10. 10Log every query, retrieved chunks, and answer. Debug real failures, not imagined ones.
APPENDIX

Glossary

ANN
Approximate Nearest Neighbor. Fast, slightly lossy search (HNSW, IVF, ScaNN). Opposite of brute-force exact search.
BM25
“Best Match 25” — the classic keyword scoring formula used by Elasticsearch.
Bi-encoder
An embedding model that encodes query and document independently into vectors. Fast, scalable, slightly less accurate than cross-encoders.
Chunk
A subsection of a document (usually 300–800 tokens) treated as the unit of retrieval.
Context window
The maximum number of tokens an LLM can read in a single prompt.
Cosine similarity
The cosine of the angle between two vectors. Ranges from -1 to +1. Most popular similarity metric for text embeddings.
Cross-encoder
A model that takes (query, document) as a pair and outputs a relevance score. Slow but very accurate; used for reranking.
dense_vector
Elasticsearch field type for storing and indexing float arrays; supports kNN queries.
Dot product
Element-wise multiply then sum. Equal to cosine similarity for unit vectors; fastest similarity to compute.
Embedding
A fixed-length vector of floats that numerically represents a piece of text; similar text produces similar vectors.
Euclidean (L2) distance
Straight-line distance between two points in space. √Σ(ai−bi)2.
Grounding
The practice of having the LLM answer from provided evidence rather than its training weights.
Hallucination
A plausible-sounding but false LLM output. RAG reduces but doesn’t eliminate this.
HNSW
Hierarchical Navigable Small World graph — the ANN algorithm behind most modern vector databases, including Elasticsearch.
Hybrid search
Combining keyword (BM25) and semantic (vector) search, usually fused with RRF.
IDF
Inverse Document Frequency. Gives rare words a bigger weight in BM25 scoring.
Inverted index
The data structure behind keyword search: a map from token → list of docs containing that token.
kNN
k-Nearest-Neighbors. Given a query vector, find the k most similar stored vectors.
LLM
Large Language Model — Claude, GPT, Llama, etc. Reads prompts, writes text.
MMR
Maximal Marginal Relevance: a reranking scheme that prefers results that are both relevant and diverse.
MTEB
Massive Text Embedding Benchmark — the standard leaderboard for comparing embedding models.
num_candidates
In Elasticsearch kNN: how many near-neighbors the HNSW search collects before returning top-k. Larger = more recall, slower.
RAG
Retrieval-Augmented Generation. Retrieve, stuff into prompt, generate.
Reranker
A second-stage, usually cross-encoder model that rescores top-K candidates from the retriever.
RRF
Reciprocal Rank Fusion — the de facto standard for fusing ranked lists from multiple retrievers. ∑1/(60+rank).
Self-attention
The mechanism inside a transformer that lets each token’s representation depend on every other token.
Semantic search
Search based on meaning (via embeddings), as opposed to keyword matching.
Token
A subword unit produced by the tokenizer. Roughly 0.75 words in English.
Transformer
The neural network architecture (2017) underlying both embedding models and LLMs.
Unit vector
A vector with length 1. Modern embedding models often normalize output to unit length.
