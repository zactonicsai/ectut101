§01 Concepts & architecture

Understand the cluster before you build one.

Elasticsearch and OpenSearch are distributed search engines. OpenSearch is an AWS-maintained fork of Elasticsearch 7.10 — the operational APIs are ~95% identical. Unless noted, everything here works for both.

node role

Master-eligible

Manages cluster state — which node holds which shard. Run exactly 3 for quorum. Two will split-brain; one has no HA.

node role

Data

Stores shards, serves search/index traffic. Scale horizontally as data grows. Needs fast disk (gp3 / Premium SSD minimum).

node role

Coordinating

Routes client requests. The Load Balancer (ALB / App Gateway / k8s Service) fronts these. Often colocated with data nodes in smaller clusters.

§02 Local simulation · Docker Compose

Simulate a 3-node cluster on your laptop.

Use this for development, index-mapping tests, and interview prep. It is not production — but the 3-node topology mirrors what you’ll deploy later.

step · 01

Prerequisites

Docker 24+, docker compose v2, 8 GB free RAM. Linux hosts must raise the vm map count:

# required on Linux, one-time per host
sudo sysctl -w vm.max_map_count=262144

§03 Kubernetes · Helm & manifests

Cluster on Kubernetes in two shapes.

Production pattern: StatefulSet + PersistentVolumes + a headless Service for peer discovery + a regular Service (or Ingress) for clients. Pick Helm for speed, raw manifests for full control.

step · 01

Add repo

helm repo add opensearch https://opensearch-project.github.io/helm-charts/
helm repo update
kubectl create namespace search
kubectl -n search create secret generic opensearch-admin \
  --from-literal=password='Str0ngPass!prod'
step · 02

values.yaml

clusterName: prod-cluster
replicas: 3
roles: [cluster_manager, data, ingest]

opensearchJavaOpts: "-Xms2g -Xmx2g"

resources:
  requests: { cpu: "1000m", memory: "4Gi" }
  limits:   { cpu: "2000m", memory: "4Gi" }

persistence:
  enabled: true
  storageClass: gp3            # AWS; 'managed-premium' on Azure
  size: 100Gi

extraEnvs:
  - name: OPENSEARCH_INITIAL_ADMIN_PASSWORD
    valueFrom:
      secretKeyRef: { name: opensearch-admin, key: password }

antiAffinity: hard               # spread pods across nodes — critical!
step · 03

Install & verify

helm -n search install opensearch opensearch/opensearch -f values.yaml

kubectl -n search get pods -w            # wait for 3/3 Ready
kubectl -n search port-forward svc/opensearch-cluster-master 9200:9200
curl -ks -u admin:'Str0ngPass!prod' https://localhost:9200/_cat/nodes?v

The raw StatefulSet below mirrors what Helm generates. Use it when you need custom tweaks, air-gap deploys, or GitOps control.

# opensearch-sts.yaml
apiVersion: v1
kind: Service
metadata: { name: opensearch-headless, namespace: search }
spec:
  clusterIP: None                       # headless for peer DNS
  selector: { app: opensearch }
  ports:
    - { name: transport, port: 9300 }
    - { name: rest,      port: 9200 }
---
apiVersion: v1
kind: Service
metadata: { name: opensearch-client, namespace: search }
spec:
  type: ClusterIP
  selector: { app: opensearch }
  ports: [{ name: rest, port: 9200, targetPort: 9200 }]
---
apiVersion: apps/v1
kind: StatefulSet
metadata: { name: opensearch, namespace: search }
spec:
  serviceName: opensearch-headless
  replicas: 3
  selector: { matchLabels: { app: opensearch } }
  template:
    metadata: { labels: { app: opensearch } }
    spec:
      initContainers:
        - name: fix-sysctl
          image: busybox:1.36
          securityContext: { privileged: true }
          command: ["sh", "-c", "sysctl -w vm.max_map_count=262144"]
      containers:
        - name: opensearch
          image: opensearchproject/opensearch:2.13.0
          env:
            - { name: cluster.name,            value: prod-cluster }
            - { name: discovery.seed_hosts,    value: "opensearch-headless" }
            - { name: cluster.initial_cluster_manager_nodes,
                value: "opensearch-0,opensearch-1,opensearch-2" }
            - { name: node.name,
                valueFrom: { fieldRef: { fieldPath: metadata.name } } }
            - { name: OPENSEARCH_JAVA_OPTS,    value: "-Xms2g -Xmx2g" }
            - name: OPENSEARCH_INITIAL_ADMIN_PASSWORD
              valueFrom:
                secretKeyRef: { name: opensearch-admin, key: password }
          resources:
            requests: { cpu: "1", memory: "4Gi" }
            limits:   { cpu: "2", memory: "4Gi" }
          ports:
            - { name: rest, containerPort: 9200 }
            - { name: transport, containerPort: 9300 }
          volumeMounts:
            - { name: data, mountPath: /usr/share/opensearch/data }
          readinessProbe:
            httpGet: { scheme: HTTPS, path: /_cluster/health, port: 9200 }
            initialDelaySeconds: 60
  volumeClaimTemplates:
    - metadata: { name: data }
      spec:
        accessModes: ["ReadWriteOnce"]
        storageClassName: gp3
        resources: { requests: { storage: 100Gi } }
kubectl apply -f opensearch-sts.yaml
kubectl -n search rollout status sts/opensearch
§04 Amazon Web Services

Two production paths on AWS.

pattern · A

Managed OpenSearch Service

AWS runs the cluster. You get a VPC endpoint. Fastest path, least control.

aws opensearch create-domain \
  --domain-name prod-search \
  --engine-version OpenSearch_2.13 \
  --cluster-config '{
    "InstanceType":"m6g.large.search",
    "InstanceCount":3,
    "DedicatedMasterEnabled":true,
    "DedicatedMasterType":"m6g.large.search",
    "DedicatedMasterCount":3,
    "ZoneAwarenessEnabled":true,
    "ZoneAwarenessConfig":{"AvailabilityZoneCount":3}
  }' \
  --ebs-options '{"EBSEnabled":true,
    "VolumeType":"gp3","VolumeSize":100}' \
  --node-to-node-encryption-options Enabled=true \
  --encryption-at-rest-options Enabled=true \
  --domain-endpoint-options EnforceHTTPS=true \
  --vpc-options 'SubnetIds=subnet-a,subnet-b,subnet-c,
    SecurityGroupIds=sg-xxx'
pattern · B

Self-managed on EKS + ALB

Full control, custom plugins, custom builds. Deploy the §03 Helm chart here, then expose via ALB Ingress.

▸ step 01 · create EKS cluster 
eksctl create cluster --name search-prod --region us-east-1 \
  --nodegroup-name search-ng --node-type m6i.xlarge \
  --nodes 3 --nodes-min 3 --nodes-max 9 \
  --with-oidc --managed
▸ step 02 · install AWS Load Balancer Controller 
helm repo add eks https://aws.github.io/eks-charts
helm install aws-load-balancer-controller \
  eks/aws-load-balancer-controller -n kube-system \
  --set clusterName=search-prod \
  --set serviceAccount.create=true \
  --set serviceAccount.name=aws-load-balancer-controller
▸ step 03 · expose via ALB Ingress 
apiVersion: networking.k8s.io/v1
kind: Ingress

§05 Microsoft Azure

Self-managed on AKS — there’s no managed OpenSearch on Azure.

Azure has Elastic Cloud marketplace integrations and Elastic’s own managed service, but no first-party managed OpenSearch. The common pattern is self-managed on AKS fronted by Application Gateway.

step · 01

AKS cluster

az group create -n search-rg -l eastus
az aks create -g search-rg -n search-aks \
  --node-count 3 --node-vm-size Standard_D4s_v5 \
  --enable-managed-identity --generate-ssh-keys \
  --zones 1 2 3
az aks get-credentials -g search-rg -n search-aks
step · 02

Install OpenSearch

Use the §03 Helm values, but change one line for Azure’s Premium SSD storage class:

persistence:
  storageClass: managed-premium      # was 'gp3' on AWS
  size: 100Gi
step · 03

App Gateway Ingress

az aks enable-addons -g search-rg -n search-aks \
  --addons ingress-appgw \
  --appgw-name search-appgw --appgw-subnet-cidr 10.225.0.0/16
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: opensearch
  namespace: search
  annotations:
    kubernetes.io/ingress.class: azure/application-gateway
    appgw.ingress.kubernetes.io/backend-protocol: https
    appgw.ingress.kubernetes.io/health-probe-path: /_cluster/health
    appgw.ingress.kubernetes.io/health-probe-status-codes: "200-401"
spec:
  rules:
  - host: search.internal.example.com

      paths:
      - path: /
        pathType: Prefix

§06 Configuring indices

Mappings, templates, lifecycle.

Always define explicit mappings for any field you’ll query seriously. Dynamic mapping is a debugging nightmare in production.

part · 01 Create an index with explicit mappings 
curl -ku admin:'Str0ngPass!prod' -X PUT \
  https://<endpoint>/orders \
  -H 'Content-Type: application/json' -d '{
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1,
    "refresh_interval": "5s",
    "analysis": {
      "analyzer": {
        "lowercase_keyword": {
          "type": "custom",
          "tokenizer": "keyword",
          "filter": ["lowercase"]
        }
      }
    }
  },
  "mappings": {
    "properties": {
      "order_id":   { "type": "keyword" },
      "customer":   { "type": "keyword", "fields": { "text": { "type": "text" } } },
      "amount":     { "type": "double" },
      "created_at": { "type": "date" },
      "items": {
        "type": "nested",
        "properties": {
          "sku": { "type": "keyword" },
          "qty": { "type": "integer" }
        }
      }
    }
  }
}'
part · 02 Index template (time-series pattern)

Any index matching the pattern inherits these settings. Essential for logs-*, orders-*, etc.

curl -ku admin:'Str0ngPass!prod' -X PUT \
  https://<endpoint>/_index_template/orders-template \
  -H 'Content-Type: application/json' -d '{
    "index_patterns": ["orders-*"],
    "template": {
      "settings": { "number_of_shards": 3, "number_of_replicas": 1 },
      "mappings": {
        "properties": {
          "order_id":   { "type": "keyword" },
          "amount":     { "type": "double" },
          "created_at": { "type": "date" }
        }
      }
    }
  }'
part · 03 Lifecycle (ISM / ILM) — rollover & delete

OpenSearch uses ISM, Elasticsearch uses ILM — concepts are identical. This policy rolls over daily and deletes after 90 days.

curl -ku admin:'Str0ngPass!prod' -X PUT \
  https://<endpoint>/_plugins/_ism/policies/orders-policy \
  -H 'Content-Type: application/json' -d '{
  "policy": {
    "description": "rollover daily, delete after 90d",
    "default_state": "hot",
    "states": [
      { "name": "hot",
        "actions": [{ "rollover": { "min_index_age": "1d", "min_size": "50gb" } }],
        "transitions": [{ "state_name": "delete",
                          "conditions": { "min_index_age": "90d" } }] },
      { "name": "delete", "actions": [{ "delete": {} }], "transitions": [] }
    ]
  }
}'
§07 Snapshot & restore

Backups are only as good as the last restore test.

take a snapshot

Manual snapshot

# all indices (wait_for_completion=false returns immediately)
curl -ku admin:'Str0ngPass!prod' -X PUT \
  "https://<endpoint>/_snapshot/s3-repo/snap-$(date +%Y%m%d-%H%M)" \
  -H 'Content-Type: application/json' -d '{
    "indices": "*",
    "ignore_unavailable": true,
    "include_global_state": false
  }'

# check progress
curl -ku admin:'Str0ngPass!prod' \
  https://<endpoint>/_snapshot/s3-repo/_all?pretty
schedule & automate

SLM / SM policy

# daily at 01:00 UTC, keep 14
curl -ku admin:'Str0ngPass!prod' -X POST \
  https://<endpoint>/_plugins/_sm/policies/daily \
  -H 'Content-Type: application/json' -d '{
    "description": "daily snapshots",
    "creation": {
      "schedule": { "cron": { "expression": "0 1 * * *",
                              "timezone": "UTC" } }
    },
    "deletion": { "condition": { "max_count": 14 } },
    "snapshot_config": { "indices": "*", "repository": "s3-repo" }
  }'
restore

Restore from snapshot

Closed or missing indices only — you cannot overwrite an open index. Close it or restore under a new name.

# close the target first (or use rename_pattern / rename_replacement)
curl -ku admin:'Str0ngPass!prod' -X POST \
  https://<endpoint>/orders/_close

curl -ku admin:'Str0ngPass!prod' -X POST \
  https://<endpoint>/_snapshot/s3-repo/snap-20260423-0100/_restore \
  -H 'Content-Type: application/json' -d '{
    "indices": "orders",
    "rename_pattern": "orders",
    "rename_replacement": "orders_restored",
    "include_aliases": false
  }'

# verify doc count matches pre-snapshot
curl -ku admin:'Str0ngPass!prod' \
  https://<endpoint>/orders_restored/_count

§09 Go indexer program

A real program you can run.

Seeds the orders index with fake documents via bulk API, then runs a search. Works against the Docker Compose sim from §02 out of the box.

file · 01

go.mod

module indexer

go 1.22

require github.com/opensearch-project/opensearch-go/v3 v3.1.0
file · 02

main.go

package main

import (
    "context"
    "crypto/tls"
    "encoding/json"
    "fmt"
    "log"
    "net/http"
    "os"
    "strings"
    "time"

    opensearch "github.com/opensearch-project/opensearch-go/v3"
    opensearchapi "github.com/opensearch-project/opensearch-go/v3/opensearchapi"
)

type Order struct {
    OrderID   string    `json:"order_id"`
    Customer  string    `json:"customer"`
    Amount    float64   `json:"amount"`
    CreatedAt time.Time `json:"created_at"`
}

func main() {
    addr := envOr("OS_ADDR", "https://localhost:9200")
    user := envOr("OS_USER", "admin")
    pass := envOr("OS_PASS", "Str0ngPass!demo")

    client, err := opensearchapi.NewClient(opensearchapi.Config{
        Client: opensearch.Config{
            Addresses: []string{addr},
            Username:  user,
            Password:  pass,
            Transport: &http.Transport{
                TLSClientConfig: &tls.Config{InsecureSkipVerify: true}, // dev only
            },
        },
    })
    if err != nil { log.Fatal(err) }

    ctx := context.Background()

    // 1. bulk-index 1000 orders
    var buf strings.Builder
    for i := 0; i < 1000; i++ {
        buf.WriteString(`{"index":{"_index":"orders"}}` + "\n")
        doc, _ := json.Marshal(Order{
            OrderID:   fmt.Sprintf("ord-%05d", i),
            Customer:  fmt.Sprintf("customer-%d", i%50),
            Amount:    10 + float64(i%500),
            CreatedAt: time.Now().UTC(),
        })
        buf.Write(doc); buf.WriteString("\n")
    }

    bulkResp, err := client.Bulk(ctx, opensearchapi.BulkReq{
        Body: strings.NewReader(buf.String()),
    })
    if err != nil { log.Fatal(err) }
    fmt.Printf("indexed %d docs in %dms, errors=%v\n",
        len(bulkResp.Items), bulkResp.Took, bulkResp.Errors)

    // 2. force refresh so docs are searchable immediately
    _, _ = client.Indices.Refresh(ctx, opensearchapi.IndicesRefreshReq{
        Indices: []string{"orders"},
    })

    // 3. search high-value orders
    query := `{
      "query": { "range": { "amount": { "gte": 400 } } },
      "size": 5,
      "sort": [{ "amount": "desc" }]
    }`
    search, err := client.Search(ctx, &opensearchapi.SearchReq{
        Indices: []string{"orders"},
        Body:    strings.NewReader(query),
    })
    if err != nil { log.Fatal(err) }

    fmt.Printf("\ntop matches: total=%d\n", search.Hits.Total.Value)
    for _, hit := range search.Hits.Hits {
        fmt.Printf("  %s → $%.2f\n", hit.ID, hit.Score)
    }
}

func envOr(k, d string) string {
    if v := os.Getenv(k); v != "" { return v }
    return d
}

best practice · shard sizing

Right-size your shards

Target: 10–50 GB per shard. Never exceed ~2B documents per shard.

Over-sharding is the single most common operational problem. A shard isn’t free — each carries file handles, Lucene segment state, and per-shard overhead. 5,000 tiny shards on a 3-node cluster will blow up the JVM before it holds any real data.

Sizing math: estimate the index’s total size at steady state, divide by ~30 GB, round up. So 600 GB / 30 = 20 primary shards. With 1 replica, that’s 40 total shards — fine on 3+ data nodes.

Rule of thumb for shard count per node: ≤ 20 shards per GB of heap. Node with 32 GB heap ⇒ cap at 640 shards.

Can’t resize after creation. The only ways to change primary count are: reindex into a new index, or use _split / _shrink (strict multiples only).

best practice · quorum

Always 3 master-eligible nodes

Why not 1? No HA. Master dies, cluster state freezes, no new shards or allocations until you bring it back.

Why not 2? Split-brain. Network partition separates them, each thinks the other is dead, both try to promote themselves, you get two clusters writing divergent state.

Why 3? Quorum is (N/2)+1 = 2. One can fail and the other two still form quorum. This is why cluster.initial_master_nodes (ES) / cluster.initial_cluster_manager_nodes (OS) must list all three names at bootstrap.

Dedicated masters in production. For clusters > 6 data nodes, run 3 separate small master-only nodes (m6g.large is plenty). They’re cheap insurance — a GC pause on a data+master node can trigger a false election.

Never run 4. Never run even numbers. 5 or 7 are fine for very large clusters where master load itself is high.

best practice · JVM heap

The 50/50 rule & 31 GB ceiling

Set Xms = Xmx. Always. Never let the JVM resize the heap — it thrashes.

50% of RAM, up to 31 GB. Linux uses the rest for filesystem cache, which Lucene depends on heavily. A cluster with 64 GB nodes running -Xms48g -Xmx48g will be slower than one running -Xms31g -Xmx31g.

Why 31 GB, not 32? Above ~32 GB the JVM disables compressed object pointers, and every object reference doubles in size. You get less effective heap with a larger heap. 31 GB is the safe ceiling.

Node bigger than 64 GB? Run two processes on it, or just use smaller instance types. Lucene loves cache; it does not love giant heaps.

kernel setting

vm.max_map_count = 262144

Lucene (the storage engine under both ES and OpenSearch) memory-maps its index files. Each segment file is one mmap region. The default Linux limit of 65,530 mmap regions per process is too low once you have real data.

If this isn’t raised, startup logs a bootstrap check failure and the node refuses to start. In Docker, Kubernetes, and WSL this setting is on the host, not inside the container.

Persist it: echo 'vm.max_map_count=262144' | sudo tee /etc/sysctl.d/99-opensearch.conf

On Kubernetes, use a privileged init container as shown in §03, or a DaemonSet that sets it on every node once.

pitfalls · compose

Docker Compose gotchas

1. Don’t use this in production. Compose has no scheduler, no self-healing, no anti-affinity. A rebooted host = all three nodes down at once.

2. Memory lock vs swap. bootstrap.memory_lock=true only works if memlock: -1 is set. Without it you’ll see GC pauses as the JVM pages to disk.

3. Volumes are per-compose. docker compose down -v wipes all data. Use docker compose down without -v unless you mean it.

4. Default password is enforced. OpenSearch 2.12+ requires OPENSEARCH_INITIAL_ADMIN_PASSWORD to be set at first boot, meeting strong-password policy, or the cluster won’t start.

5. Self-signed TLS. The demo certs are trust-all. Never expose port 9200 outside localhost on a compose setup.

best practice · scheduling

Why antiAffinity=hard

Without a hard pod anti-affinity rule, Kubernetes may schedule all three cluster pods onto the same node (especially on small clusters or during evictions). When that node reboots, you lose the entire cluster — primary and replica shards, at the same time.

Hard vs soft:

  • hard = requiredDuringSchedulingIgnoredDuringExecution — pod stays Pending until it can be placed on a different node. Safe.
  • soft = preferredDuringScheduling — scheduler tries, gives up if no options. Risky.

Even better in prod: topology spread across availability zones using topologySpreadConstraints with topology.kubernetes.io/zone. Combined with OpenSearch’s cluster.routing.allocation.awareness.attributes: zone, replicas are guaranteed in different AZs.

gotchas · ALB

AWS ALB in front of OpenSearch

success-codes must include 401. If security is enabled, /_cluster/health returns 401 without auth. The ALB marks targets unhealthy unless you accept 401 as healthy. success-codes: "200,401".

backend-protocol: HTTPS. OpenSearch speaks TLS on 9200 in secure mode. Don’t terminate TLS at the ALB and talk plaintext backend unless you’ve explicitly disabled the security plugin.

target-type: ip. Use IP mode (not instance) when pods are behind a CNI like AWS VPC-CNI — ALB talks directly to pod IPs, no extra NAT hop.

scheme: internal. Never expose the cluster to the public internet. Use an internal ALB + private Route53 zone + VPN / client VPN for operator access.

ALB has a 60s idle timeout. Long-running tasks (reindex, snapshot status polling) will get chopped. Use idle_timeout.timeout_seconds annotation to bump it, or use wait_for_completion=false on long operations.

mapping · pitfalls

keyword vs text — they are not the same

text — analyzed. String is tokenized, lowercased, stemmed. Good for full-text search ("quick brown fox" → ["quick","brown","fox"]). Bad for exact matches, aggregations, sorting.

keyword — not analyzed. Stored as-is. Good for exact match, terms aggregations, sorting, filtering. Bad for free-text search.

The common trick — map both, as in the customer field in §06. The outer field is keyword (for filtering/aggregation), with a sub-field .text for free-text search. Query both where you need both.

Don’t rely on dynamic mapping for IDs. A numeric-looking ID like "12345" gets mapped as long, and then "12345A" breaks everything. Always declare IDs as keyword explicitly.

pattern · tiering

Hot-warm-cold data tiering

Most time-series workloads have 80% of queries hitting the most recent 20% of data. Tiering exploits that with cheaper storage for older indices.

  • hot — latest 7 days, nvme / gp3, high replica count, actively indexed.
  • warm — 7–30 days, gp3 or throughput-optimized HDD (st1), replicas = 1, read-mostly. Shrink + forcemerge before transition.
  • cold — 30–90 days, local st1 or searchable snapshots on S3, replicas = 0.
  • delete — after retention horizon.

Implement via ISM state transitions with index.routing.allocation.require.box_type: warm + node attributes node.attr.box_type=warm.

thresholds · disk

Disk watermarks — 85 / 90 / 95

The cluster auto-reacts to disk pressure via three watermarks:

  • low (85%) — no new shards allocated to this node.
  • high (90%) — tries to move shards away from this node. More disk pressure and more rebalance traffic.
  • flood (95%) — index blocks activate. All indices with shards on this node become read-only. This is usually what takes you down.

Alert at 75% — you need 10% of headroom to resolve before watermarks trigger. By the time the low watermark fires, it’s already too late to avoid rebalance.

Recovery from flood: PUT _all/_settings {"index.blocks.read_only_allow_delete":null} then add disk / delete old indices.

DR drill checklist

Restore from backup — best practices

  1. Test restore quarterly. An untested backup is a claim, not a backup.
  2. Restore to a separate cluster. Never test by restoring over production. Spin up a temporary cluster, restore, count docs, run sample queries, tear down.
  3. Rename on restore. Use rename_pattern / rename_replacement so you don’t conflict with live indices.
  4. Verify doc counts, not just “green”. _count + sample _search for known records.
  5. Watch version compatibility. You cannot restore a snapshot into a cluster more than one major version older than the source. Restoring a 2.x snapshot into a 1.x cluster will fail.
  6. Repository credentials are per-cluster. A new cluster restoring from the same S3 bucket needs the same IAM role / access keys registered.
  7. Record your RTO/RPO. Daily snapshot + 30 minute restore = RPO ≤ 24h, RTO ≈ 30min. Document it.
production · hardening

Hardening the Go client for production

  1. Drop InsecureSkipVerify. Load the cluster CA into RootCAs. Never trust-all in production.
  2. Use the bulk indexer helper, not a raw string builder. opensearchutil.NewBulkIndexer handles batching, backpressure, and retry.
  3. Set timeouts. Every request needs a context.WithTimeout. The client default is zero == forever, which will hang goroutines on a bad node.
  4. Retry with backoff on 429 (too many requests) and 5xx. Don’t retry 4xx.
  5. Set a connection pool size (MaxIdleConnsPerHost) proportional to concurrency. Default 2 will bottleneck.
  6. Instrument it. Emit Prometheus metrics for bulk batch size, bulk errors, search latency p50/p95/p99.
  7. Don’t refresh after every write. refresh=true is an antipattern in hot paths — use the index’s refresh_interval setting (default 1s).
