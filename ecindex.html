1) Overview: what Elasticsearch is really doing

Elasticsearch stores JSON documents inside an index. That index has mappings that define field behavior and settings that control operations like shard and replica layout. When you load data, Elasticsearch analyzes fields, writes documents into shard segments, and later fans search queries out across shards before merging the results.

Document

One JSON object you want to search, such as a product, article, case file, or log entry.

Mapping

Defines whether a field is full-text searchable, exact-match only, numeric, date-based, and more.

Shard layout

Determines how data is split and duplicated, which directly affects indexing, resilience, and search speed.

2) Diagram: how indexing and searching connect

The most important idea in Elasticsearch is that your field design controls your search experience. Data is not just stored. It is transformed into searchable structures.

JSON Documents
title, category, date, description
Mapping + Analysis
text, keyword, float, date
Primary Shards
routing, indexing, segment storage
User Query
match, term, range
Search Fan-Out
each shard searched in parallel
Merged Results
top hits, scores, filters, sorting
Key takeaway: mapping affects what gets indexed, shards affect how it is distributed, and queries behave differently depending on both.

3) Lab setup

This lab assumes Elasticsearch is reachable at http://localhost:9200.

Option A: quick local Docker run

docker run --name es01 \
  -p 9200:9200 -p 9300:9300 \
  -e "discovery.type=single-node" \
  -e "xpack.security.enabled=false" \
  docker.elastic.co/elasticsearch/elasticsearch:9.1.0

Check that Elasticsearch is up

curl http://localhost:9200

You should see cluster information in JSON.

Why start single-node?

It removes cluster complexity and lets you focus on the core ideas: mappings, indexing, querying, and shard behavior. In a single node lab, setting replicas to 0 is often practical because there is nowhere else to place a replica.

4) Create your first index the right way

Do not begin by dumping random JSON into Elasticsearch and hoping dynamic mapping guesses correctly. Create the index deliberately. Decide which fields are for full-text search and which are for exact matching, sorting, filtering, or aggregations. 

curl -X PUT "http://localhost:9200/books" \
  -H "Content-Type: application/json" \
  -d '{
    "settings": {
      "number_of_shards": 1,
      "number_of_replicas": 0
    },
    "mappings": {
      "properties": {
        "title": {
          "type": "text",
          "fields": {
            "raw": { "type": "keyword" }
          }
        },
        "author":      { "type": "keyword" },
        "published":   { "type": "date" },
        "price":       { "type": "float" },
        "category":    { "type": "keyword" },
        "description": { "type": "text" },
        "in_stock":    { "type": "boolean" }
      }
    }
  }'

Why this mapping?

  • title is full-text searchable
  • title.raw supports exact sort/filter use cases
  • author and category are exact-match fields
  • published and price support range queries
  • in_stock makes boolean filters cheap and clean

Important

Primary shard count is a design-time choice. You can change replicas later much more easily than primaries.

Why Bulk matters

Every HTTP request has overhead. Bulk reduces that overhead by batching many actions into one request. That increases ingestion throughput and better matches how Elasticsearch is optimized internally.

6) Search and filter

Full-text search and exact filters are different. That is why text and keyword exist as separate field strategies.

Full-text query

curl -X GET "http://localhost:9200/books/_search" \
  -H "Content-Type: application/json" \
  -d '{
    "query": {
      "match": {
        "description": "indexing search"
      }
    }
  }'

Why it works: description is a text field, so Elasticsearch analyzes both stored content and query input.

Exact filter

curl -X GET "http://localhost:9200/books/_search" \
  -H "Content-Type: application/json" \
  -d '{
    "query": {
      "term": {
        "category": "search"
      }
    }
  }'

Why it works: category is a keyword field, so Elasticsearch matches the exact stored value.

Range query example

curl -X GET "http://localhost:9200/books/_search" \
  -H "Content-Type: application/json" \
  -d '{
    "query": {
      "range": {
        "price": {
          "gte": 40,
          "lte": 60
        }
      }
    }
  }'

7) Shards and replicas without confusion

A primary shard stores one partition of your index. A replica shard is a copy of a primary. More shards are not automatically better. Too many shards create metadata, coordination, and resource overhead.

Inspect index info

curl "http://localhost:9200/_cat/indices?v"

Inspect shard placement

curl "http://localhost:9200/_cat/shards/books?v"
Choice Helps with Tradeoff
More primary shards Distribution and parallelism More overhead and coordination
More replicas Read scale and resilience More disk and write cost
Few large shards Lower coordination cost Can limit scale and recovery flexibility

Simple beginner rule

For a small project, start with 1 primary shard. Add complexity only when data size, write rate, or search load proves that you need it.

8) Best practices that improve search results and operations

Map fields intentionally

Do not let dynamic mapping make all important decisions for production data.

Separate search from filtering

Use text for full-text search and keyword for exact matching, sorting, and aggregations.

Start with fewer shards

Over-sharding is one of the most common beginner mistakes.

Use Bulk for ingestion

Bulk indexing reduces request overhead and improves throughput.

Normalize data

Consistent date formats, clean categories, and stable IDs make search more predictable.

Test query behavior early

Load a small sample, run real queries, then refine mappings before scaling up.

Common mistakes

  • Making every string field text
  • Using too many shards too early
  • Indexing large datasets one document at a time
  • Ignoring exact-match and filter needs
  • Trying to redesign mappings after millions of documents are loaded

Why indexes matter

An index is more than a folder of JSON. It is the searchable structure that Elasticsearch builds from your field mappings and settings. If the index is designed well, your search queries feel fast and accurate. If it is designed poorly, you may get bad relevance, slow filters, and scaling headaches later.

Text vs keyword

Use text when people search human language. Use keyword when values must match exactly, such as IDs, categories, tags, emails, status codes, and other structured values. A common pattern is to map one field both ways using multi-fields.

Mapping explainer

Mapping is how you tell Elasticsearch what each field means. That matters because a match query on text behaves very differently from a term query on keyword. Your mapping choices influence tokenization, filtering, sorting, aggregation behavior, and query relevance.

Shards explainer

Elasticsearch distributes an index across primary shards. A replica is an additional copy of a primary shard. Searches can run against shard copies, which helps with read scale and resilience. But every shard also costs memory, metadata, and coordination overhead, so over-sharding can hurt performance.

Query types explained

  • match: full-text query for analyzed text
  • term: exact-match query for structured values
  • range: query numeric or date intervals

Why relevance changes

Relevance depends on field analysis, token matching, and scoring across the indexed terms in your shard segments. If a field is mapped poorly, relevance will feel wrong because Elasticsearch built the wrong kind of searchable structure for that field.
