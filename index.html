CHAPTER 01

The architecture at a glance

Six containers on one Docker network. The UI talks to the API. The API talks to whichever Elasticsearch node answers first. The Go watchdog watches both nodes. Kibana gives you a visual debugger.

01
Single network

Every container joins eslab. They reach each other by service name (http://es01:9200) — no IPs, no DNS config.

02
Two ES nodes, equal role

Both nodes are master-eligible AND data. That’s fine for a lab; in production you split roles to isolate master election from data load.

03
Shard layout

2 primaries + 1 replica each ⇒ 4 shards total, two per node, every primary mirrored on the other node. Kill one, you lose zero data.

CHAPTER 02

docker-compose.yml · line by line

The orchestration file. Every line here is a decision: about isolation, about data persistence, about how services find each other.

2.1   File header & version

01# =============================================================================
02# Elasticsearch Learning Lab
03# Two-node ES cluster + Kibana + FastAPI + Go watchdog + static UI
04# =============================================================================
05version: "3.8"
WHAT

version: "3.8" declares the Compose file schema. 3.8 is the last of the V3 line and supports healthchecks, long-form volumes, and the features we’ll use below.

WHY

Compose V2 (the docker compose CLI, not the older docker-compose) technically doesn’t need the version key anymore — it’s ignored. But keeping it documents your intent and makes the file portable to older hosts.

BETTER

In a real repo, replace every literal value (image tag, password, memory) with ${VAR:-default} substitution and ship a .env.example. That lets CI, local dev, and prod all run from the same file with different env files.

2.2   The first Elasticsearch node

06services:
07  es01:
08    image: docker.elastic.co/elasticsearch/elasticsearch:8.13.4
09    container_name: es01
10    environment:
11      - node.name=es01
12      - cluster.name=lab-cluster
13      - discovery.seed_hosts=es02
14      - cluster.initial_master_nodes=es01,es02
15      - bootstrap.memory_lock=true
16      - xpack.security.enabled=false
17      - "ES_JAVA_OPTS=-Xms512m -Xmx512m"
WHAT · line 08

The official Elastic image from their registry. Pinning 8.13.4 (not :latest) means every teammate and every CI run gets bit-for-bit the same binary.

WHY pin?

An ES minor upgrade can change default settings, deprecate APIs, or add new required configs. Never let a latest tag silently upgrade your lab between two runs of the same tutorial.

WHAT · line 11

node.name=es01 — the human-readable identifier this node reports to the cluster. Appears in _cat/nodes and in every log line.

WHY explicit?

Without it ES auto-generates a random node name. Then when you’re debugging a shard allocation issue you’re staring at VeV4vfLK instead of es01. Always name nodes.

WHAT · line 12

cluster.name=lab-cluster — the shared identifier. All nodes with the same name form one cluster.

GOTCHA

If two nodes disagree on cluster.name, they each think they’re the only one alive. You’ll see two single-node clusters with no errors in the log. Always grep cluster.name across all nodes when troubleshooting discovery.

WHAT · line 13

discovery.seed_hosts=es02 — a list of peers this node should try to contact at startup to find the rest of the cluster.

WHY just es02?

Because Docker Compose gives every service a DNS entry matching the service name. Inside the eslab network, es02 resolves to the es02 container’s IP. No hardcoded IPs, no config updates when containers recycle.

WHAT · line 14

cluster.initial_master_nodes=es01,es02 — the bootstrapping seed. The very first time the cluster forms, exactly these nodes must vote to elect the first master.

DO NOT skip

Missing this on a fresh cluster = it never forms. You’ll see endless master not discovered yet in logs. After the first successful bootstrap the setting is ignored, but you can’t start without it the first time.

WHAT · line 15

bootstrap.memory_lock=true — tells the JVM to lock its heap into physical RAM. The kernel is forbidden from swapping it to disk.

WHY critical

If any part of the ES heap gets paged out, GC pauses spike from milliseconds to seconds. The cluster declares the node dead, shards reallocate, the flood of recovery traffic takes down other nodes. This is the classic “death spiral.”

WARNING · line 16

xpack.security.enabled=false disables authentication, TLS, role-based access, audit logs — the entire security layer. Only acceptable on localhost learning labs.

BETTER · production

Always true. Then mount TLS certs, set ELASTIC_PASSWORD, and configure TLS for both the HTTP and transport layers. See chapter 10 for the exact production block.

WHAT · line 17

ES_JAVA_OPTS=-Xms512m -Xmx512m — JVM minimum and maximum heap, both forced to 512 MB.

WHY Xms == Xmx

If the JVM can resize the heap, every resize is a stop-the-world event. Freezing it at one size means fewer surprises. 512 MB is a laptop-sized lab value; production is 50% of host RAM up to 31 GB (above that the JVM disables compressed pointers and you actually lose effective heap).

2.3   ulimits, volumes, ports, health

18    ulimits:
19      memlock: { soft: -1, hard: -1 }
20      nofile:  { soft: 65536, hard: 65536 }
21    volumes:
22      - es01-data:/usr/share/elasticsearch/data
23    ports:
24      - "9200:9200"
25    networks: [eslab]
26    healthcheck:
27      test: ["CMD-SHELL", "curl -fs http://localhost:9200/_cluster/health | grep -E '\"status\":\"(green|yellow)\"'"]
28      interval: 10s
29      timeout: 5s
30      retries: 12
WHAT · lines 18-20

memlock: -1 means “unlimited.” This is the companion to bootstrap.memory_lock=true — the JVM can try to lock memory, but without the kernel raising RLIMIT_MEMLOCK for the container, the lock call silently fails. Both settings are required, not one or the other.

nofile: 65536 lifts the max-open-files limit. Lucene (the library ES is built on) holds a file descriptor for every segment file. A busy index easily breaks the default 1024 limit.

WHAT · line 22

Named volume es01-data mounted at /usr/share/elasticsearch/data. This is where ES writes every shard, every segment, every translog.

WHY named, not bind

Named volumes are managed by Docker and survive container rebuilds. A bind mount (./data:/usr/share/...) drags your host filesystem quirks (permissions, SELinux, APFS slowness on macOS) into the container. ES is I/O-heavy — you want a real Docker volume.

AWS EQUIVALENT

On EKS this becomes a PersistentVolumeClaim backed by an EBS gp3 volume, provisioned via the EBS CSI driver. On ECS Fargate it’d be an EFS mount (though Fargate is generally a poor fit for stateful ES workloads). On AWS Managed OpenSearch the entire storage layer is opaque — you just declare volume size.

WHAT · line 24

"9200:9200" publishes the container’s REST port to the host. You can curl http://localhost:9200 from outside Docker.

PRODUCTION

Never publish 9200 from a production ES node. All client traffic should go through a load balancer or coordinating-only nodes. Nodes themselves communicate on port 9300 (the transport protocol) which should stay internal.

WHAT · lines 26-30

The healthcheck asks the node for its cluster status every 10 seconds. A node is "healthy" only when /_cluster/health returns "status":"green" or "yellow".

The API container depends on this — Compose waits for es01 to be healthy before starting the API.

WHY accept yellow?

Yellow means "primaries are fine, some replicas are still initializing." That’s a perfectly valid state to serve reads and writes. Requiring green would make the API wait forever on a fresh cluster that has no replicas assigned yet.

The remaining services

es02 is a near-mirror of es01. The interesting parts are Kibana, the API build, the watchdog, and the static-file UI — each wired together with depends_on.

3.1   es02 — what changes, what doesn’t

35  es02:
36    image: docker.elastic.co/elasticsearch/elasticsearch:8.13.4

WHAT

Three values are different from es01, everything else is identical. The port mapping is 9201:9200 — the container still listens internally on 9200, but we expose it on host port 9201 to avoid colliding with es01.

WHY symmetrical seed_hosts

es01 seeds from es02, es02 seeds from es01. Either can start first and still discover its partner. The fragile alternative — having both point at a single "primary" — breaks when that one node is down at boot.

3.2   Kibana

55  kibana:
56    image: docker.elastic.co/kibana/kibana:8.13.4

WHAT · line 58

Kibana accepts a list of ES hosts. It will round-robin read requests and transparently fail over if one is down.

WHY both nodes

If you list only es01 and es01 dies, Kibana shows a blank page even though half the cluster is fine. The full list is a one-line resilience win.

WHAT · lines 60-61

depends_on: condition: service_healthy means Compose will not start Kibana until the es01 healthcheck passes. Without it, Kibana starts first, fails to connect, and prints a scary banner even though ES is coming up normally.

The Python code splits on comma and hands the list to the Elasticsearch(...) constructor. Same pattern as Kibana — if one node is down, the client tries the next. This is client-side failover, no proxy involved.

3.4   api/Dockerfile

01FROM python:3.12-slim
02WORKDIR /app
03COPY requirements.txt .
04RUN pip install --no-cache-dir -r requirements.txt
05COPY . .
06EXPOSE 8000
07CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
WHAT · line 01

python:3.12-slim is the Debian-slim variant — ~45 MB vs ~900 MB for the full image. Contains Python + pip and little else.

WHY slim

Smaller images = faster CI, smaller attack surface, less storage on production nodes. Only go with the full python:3.12 if you need build tools (gcc, headers) to compile extensions; we don’t here.

BETTER

For real production, use a multi-stage build or python:3.12-slim + pip install --user + COPY --from=builder to drop build deps from the final layer. Also run as non-root with USER 1000, and add HEALTHCHECK.

WHAT · lines 03-05

Two COPY statements: requirements first, then everything else. Reason: Docker caches each layer. As long as requirements.txt doesn’t change, the pip install layer is reused, and code-only changes produce tiny rebuilds.

WHAT · line 07

uvicorn is the ASGI server that actually runs FastAPI. main:app = import app from main.py. --host 0.0.0.0 is required for container networking — the default 127.0.0.1 would only accept traffic from within the container.

3.5   Static UI with nginx

79    image: nginx:alpine
80    container_name: ui
81    volumes:
82      - ./ui:/usr/share/nginx/html:ro
83    ports: ["8080:80"]
WHAT

No build step. Bind-mount the ui/ directory directly into nginx’s default docroot, read-only. Nginx serves the single index.html.

WHY no build

Tailwind is loaded from a CDN in the HTML (<script src="https://cdn.tailwindcss.com">). Zero npm, zero bundler, zero build. That’s fine for a teaching artifact; for production you’d compile Tailwind with PurgeCSS to ship only the classes you use (~10 KB instead of ~300 KB).

AWS EQUIVALENT

Replace nginx with an S3 bucket + CloudFront distribution. Upload index.html, point CloudFront at the bucket, done. Cheaper, globally cached, zero servers to manage. This is the #1 "move to AWS" win — static sites shouldn’t run on VMs in 2026.

3.6   Go watchdog

86  watchdog:
87    build: ./go-watchdog
88    container_name: watchdog
89    environment:
90      - ES_NODES=http://es01:9200,http://es02:9200
91      - POLL_SECONDS=3
WHAT

A separate container running a custom Go binary that polls both ES nodes every 3 seconds and logs state transitions. Full walkthrough in Chapter 07.

WHY a separate process

Failover detection is a cross-cutting concern — nothing to do with business logic. Putting it in the API would mean the detector dies when the API dies. A separate tiny process can page on-call, update a registry, or trigger auto-recovery scripts even if every other service is broken.

AWS EQUIVALENT

On AWS Managed OpenSearch you get this for free: CloudWatch metrics + CloudWatch Alarms + EventBridge. For self-managed on EKS, replace this container with Prometheus + Alertmanager scraping the elasticsearch_exporter.

api/main.py · the FastAPI backend

This file owns every Elasticsearch interaction. Four things matter here: how we connect, how we build mappings, how we handle documents, and how we build the search query.

4.1   Imports & the client

WHAT · line 17

Read ES_NODES from the environment, fall back to localhost, split on comma. End result is a Python list like ["http://es01:9200", "http://es02:9200"].

WHY parameterize

Same code, same image, different targets in dev/staging/prod. The 12-factor app principle: config lives in the environment, never in the source tree. A hardcoded URL means you can’t promote the same image up the environment ladder.

WHAT · lines 20-25

Construct one Elasticsearch client for the process lifetime. It accepts the list of nodes — internally it maintains a connection pool, tracks which nodes are reachable, and marks dead nodes so new requests skip them.

WHY one global client

The client maintains a persistent HTTP connection pool. Re-creating it per request means a fresh TCP handshake + TLS negotiation every time — 10× slower. FastAPI runs multiple worker processes; each gets its own client and its own pool, which is exactly right.

WHAT · lines 22-24
  • request_timeout=10 — if a request hangs more than 10 seconds, raise.
  • max_retries=3 — automatically retry transient failures (connection errors, timeouts).
  • retry_on_timeout=True — include timeouts in what counts as retryable.
BETTER

Add sniff_on_start=True and sniff_on_node_failure=True so the client periodically re-discovers all nodes from the cluster, picking up new ones added after process start. And tune maxsize (connection pool size) to match your uvicorn worker concurrency.

4.2   CORS middleware

36    allow_origins=["*"],

WHAT

Cross-Origin Resource Sharing middleware. The UI loads from :8080 (nginx) and calls the API on :8000 — different origin, so the browser blocks the request unless the server explicitly allows it.

NEVER in production

allow_origins=["*"] is a development convenience. In production list every exact origin that should be allowed: ["https://search.example.com"]. A wildcard means any site on the internet can make authenticated requests to your API.

BETTER

Skip CORS entirely by serving the UI from the same origin as the API — put a reverse proxy (nginx, or the ALB in front) that routes /api/* to the FastAPI container and everything else to the static UI. Same-origin is simpler and more secure than configuring CORS right.

4.3   Pydantic request models

46class IndexSpec(BaseModel):
47    name: str = Field(..., description="Index name, e.g. 'articles'")
48    shards: int = Field(2, ge=1, le=10)
49    replicas: int = Field(1, ge=0, le=5)
50    keyword_fields: list[str] = Field(default_factory=list)
51    text_fields: list[str] = Field(default_factory=lambda: ["content", "title"])
WHAT

Pydantic declares the request body. FastAPI auto-parses incoming JSON, validates types, enforces the bounds (ge=1, le=10), and rejects malformed requests with a structured 422 response. And it generates OpenAPI docs at /docs from these annotations for free.

WHY bounds on shards

Two guardrails: someone can’t ask for 10,000 shards and explode the cluster, and they can’t ask for 0 shards (invalid). The UI could validate this client-side but a server-side guardrail is non-negotiable — never trust input from the browser.

4.4   The mapping builder — the heart of the API

70def build_mapping(spec: IndexSpec) -> dict:
71    props: dict[str, dict] = {
72        "ingested_at": {"type": "date"},
73    }
74    for f in spec.text_fields:
75        props[f] = {
76            "type": "text",
77            "fields": {"keyword": {"type": "keyword", "ignore_above": 256}}
78        }
79    for f in spec.keyword_fields:
80        props[f] = {"type": "keyword"}
81    return {
82        "settings": {
83            "number_of_shards": spec.shards,
84            "number_of_replicas": spec.replicas,
85            "refresh_interval": "1s",
86        },
87        "mappings": {"properties": props},
88    }
WHAT · lines 74-78 — the sub-field trick

Every text field gets mapped as analyzed text, plus a nested .keyword sub-field that stores the raw value unchanged (up to 256 chars).

WHY this matters

Text fields can be full-text searched (match query) but can’t be used for aggregations, sorting, or exact filters — they’re tokenized. The sub-field gives you both behaviors: query title for fuzzy search, query title.keyword when you want to group by exact title.

ignore_above: 256 means strings longer than 256 chars aren’t indexed in the keyword sub-field. Protects against blowing up memory on a 10 MB text blob.

WHAT · line 85

refresh_interval: 1s — how often ES makes newly-indexed docs searchable. 1 second is the default and means "near-real-time search."

BETTER for bulk loads

During a giant initial import (millions of docs), set refresh_interval: -1 and number_of_replicas: 0, run the bulk, then put both settings back. 3-5× throughput improvement on first load because ES doesn’t build search segments or write replicas as you go.

4.5   File upload endpoint

140@app.post("/documents/upload", status_code=201)
141async def upload_text_file(
142    file: UploadFile = File(...),
143    index: str = Form(INDEX_NAME),
144    tags: str = Form(""),
145    author: str = Form("unknown"),
146):
147    raw = (await file.read()).decode("utf-8", errors="replace")
148    if not es.indices.exists(index=index):
149        es.indices.create(index=index, body=build_mapping(...))
150    body = {
151        "title": file.filename,
152        "content": raw,
153        "tags": [t.strip() for t in tags.split(",") if t.strip()],
154        ...
155    }
156    es.index(index=index, id=doc_id, document=body, refresh="wait_for")
WHAT · line 147

Read the uploaded file into memory and decode as UTF-8. errors="replace" means invalid bytes become U+FFFD (the replacement character) rather than throwing. Safe default for unknown-source text.

MEMORY LIMIT

await file.read() loads the entire file into RAM. A 500 MB log file would OOM the container. For anything larger than a few MB, stream with await file.read(chunk_size) in a loop or use the ES bulk helpers to index line by line.

WHAT · line 156

refresh="wait_for" means the write request doesn’t return until the doc is searchable. Up to 1 second delay, but the UI can search immediately after upload.

Three refresh modes
  • refresh=False (default) — return immediately, doc becomes searchable within refresh_interval. Fastest for bulk.
  • refresh="wait_for" — return after the next natural refresh. Good balance for interactive writes.
  • refresh=True — force an immediate refresh. Never use in hot paths — it generates a tiny segment per write and hurts performance cluster-wide.
CHAPTER 05

Why text and keyword are different

This one mapping decision determines whether your queries work. If you get it wrong, you’ll hit it again at 2 AM when a field you thought was searchable isn’t.

Use "text" for
  • • Body content, article text, product descriptions
  • • Anywhere users type free-form queries
  • • Fields where fuzzy matching ("foxes" finds "fox") is desired
Use "keyword" for
  • • IDs, SKUs, email addresses, hostnames
  • • Tags, categories, enum-like values
  • • Anything you’ll filter on (author:alice)
  • • Anything you’ll aggregate on (count by category)
  • • Anything you’ll sort by alphabetically
CLASSIC BUG

You let ES dynamically map a field called status. Docs come in with values like "PENDING", "COMPLETE". ES chooses type=text, lowercases everything. Now {"term": {"status": "PENDING"}} returns zero results because the tokens stored are pending, not PENDING. Declare mappings explicitly. Never rely on dynamic mapping for production.

CHAPTER 06

The search endpoint — query construction

This is the endpoint users hit. It translates three simple query parameters into a real Elasticsearch Query DSL query. 

186@app.get("/search")
187def search(
188    q: str = "",                           # free-text query
189    field: Optional[str] = None,            # restrict to one field
190    keyword: Optional[str] = None,          # exact "field:value"
191    index: str = INDEX_NAME,
192    size: int = 10,
193):
194    must: list[dict] = []
195
196    if q:
197        if field:
198            must.append({"match": {field: q}})
199        else:
200            must.append({
201                "multi_match": {
202                    "query": q,
203                    "fields": ["title^3", "content", "*"],
204                    "fuzziness": "AUTO",
205                }
206            })
207
208    if keyword and ":" in keyword:
209        k, v = keyword.split(":", 1)
210        must.append({"term": {k: v}})
211
212    query = {"match_all": {}} if not must else {"bool": {"must": must}}
213
214    result = es.search(
215        index=index, query=query, size=size,
216        highlight={"fields": {"content": {}, "title": {}}},
217    )
WHAT · lines 194-212

We build a bool query by collecting clauses into a must list. Empty list → match_all (return everything). Non-empty → wrap in bool.must which ANDs the clauses.

WHY build it this way

Three API parameters, each optional. You can pass all, any two, or just one. Constructing the query as a list of clauses and joining them means the code stays linear — we don’t need a nested if/else for every combination. This pattern scales to 10+ filters.

WHAT · line 203

"fields": ["title^3", "content", "*"] — search across title (weighted 3×), content (1×), and every other field ("*").

WHY weight title

A match in the title is a much stronger signal than a match buried in the body. The ^3 suffix tells the BM25 scorer to triple the score contribution from title matches — your most relevant docs bubble to the top.

WHAT · line 204

"fuzziness": "AUTO" enables typo tolerance. Search "elasticsrch" and it will still find "Elasticsearch".

AUTO is smart

AUTO means no fuzziness on 1-2 character terms (they’d match everything), 1 edit allowed on 3-5 char terms, 2 edits on 6+ char terms. Specifying a fixed "fuzziness": 2 would let a 3-character query match anything — destroying precision.

WHAT · line 210

term (not match) for keyword filter. term looks up the exact value in the inverted index without running it through any analyzer.

term vs match

Using match on a keyword field happens to work but is semantically wrong — match would run the query through an analyzer first (for keyword fields that’s a no-op, so it works by accident). Use term for keywords, match for text. Never conflate them.

WHAT · line 216

highlight tells ES to return the matched fragments with <em> tags around hit terms. The UI renders these with yellow highlighting. Essentially free — it’s computed from data ES already has.

CHAPTER 07

The Go watchdog — 60 lines that teach state machines

No third-party deps. Just a loop, a map, and an HTTP client. It exists to demonstrate two things: how to detect failure, and why you should detect it in a separate process.

7.1   The probe function

42var httpClient = &http.Client{Timeout: 2 * time.Second}
43
44func probe(url string) (nodeState, *clusterHealth) {
45    st := nodeState{URL: url}
46    resp, err := httpClient.Get(url + "/_cluster/health")
47    if err != nil {
48        st.LastErr = err.Error()
49        return st, nil
50    }
51    defer resp.Body.Close()
52    if resp.StatusCode != 200 {
53        st.LastErr = fmt.Sprintf("http %d", resp.StatusCode)
54        return st, nil
55    }
56    var h clusterHealth
57    json.NewDecoder(resp.Body).Decode(&h)
58    st.Up = true
59    return st, &h
60}
WHAT · line 42

Package-level httpClient with a 2-second total timeout.

TIMEOUT TRAP

Go’s default http.Client has no timeout (it hangs forever). Production services that forget to set one eventually have goroutines stacked on dead connections. Always set it. 2s is right here — the probe runs every 3s, so longer timeouts queue up.

WHAT · line 46

Hit /_cluster/health. Any 2xx response means the node is reachable AND knows its cluster state. A response from a healthy-network node that just crashed internally would be an error or a 5xx — we’d catch it.

WHAT · line 51

defer resp.Body.Close() is not optional in Go. If you don’t close the body, the underlying TCP connection stays open and can’t be reused. Eventually the client leaks FDs and new requests fail.

7.2   The transition detector

90    prev := make(map[string]nodeState, len(nodes))
91    for _, u := range nodes {
92        prev[u] = nodeState{URL: u, Up: false}
93    }
94    prevClusterStatus := ""

96    tick := time.NewTicker(time.Duration(pollSec) * time.Second)

99            curr, h := probe(u)
100            before := prev[u]
101            switch {
102            case !before.Up && curr.Up:
103                log.Printf("✅ RECOVERED  %s", u)
104            case before.Up && !curr.Up:
105                log.Printf("❌ FAILED     %s", u)
106            }
107            prev[u] = curr
108        }
109    }
WHAT · lines 90-93

Seed every node as Up: false. On the first iteration, every reachable node will be logged as RECOVERED — that’s fine, that’s the initial "coming online" signal.

WHAT · lines 101-106 — the state machine

Only four possible transitions:

  • false → true : RECOVERED
  • true → false : FAILED
  • true → true : still up (silent)
  • false → false : still down (silent)

We only log on a change. Otherwise the log floods with every poll.

WHY this pattern matters

Edge-triggered monitoring. An alerting system that pages on "the node is still down" after the 100th poll wakes you up 99 times for the same thing. Edge-triggered alerts fire once per state change. This is the foundation of every alerting rule in Prometheus, PagerDuty, CloudWatch.

BETTER for production

Three hardenings:

  1. Flap suppression — require 3 consecutive failed probes before declaring FAILED. Single blips shouldn’t page.
  2. Webhook out — POST to PagerDuty / Slack / Opsgenie on transitions, not just log.
  3. Expose Prometheus metrics — run an HTTP server on :9100 with elasticsearch_node_up gauge. Then Alertmanager handles the flap suppression, routing, and escalation for you.
AWS EQUIVALENT

On AWS Managed OpenSearch you get CloudWatch metrics out of the box: ClusterStatus.green, ClusterStatus.yellow, ClusterStatus.red. Attach a CloudWatch Alarm with a SNS topic target, and SNS fans out to email/PagerDuty. The watchdog becomes one Alarm resource in Terraform.

9.1   What the stack looks like on AWS

THE KEY DIFFERENCE

On AWS, you stop owning the failure modes of the data plane. With managed OpenSearch, AWS runs the nodes, replaces failed hardware, applies security patches, handles master election, and takes snapshots to S3 without you lifting a finger. Your watchdog isn’t deleted — it becomes a CloudWatch Alarm. Your volumes aren’t visible — they just work. You go from operating a cluster to consuming it.

The trade-off: you can’t install arbitrary plugins, you’re ~6-12 months behind the latest OpenSearch release, and you pay a 20-30% premium over raw EC2. For ~90% of workloads, that’s the right trade.

CHAPTER 10

Production checklist

What must change before this stack carries a single real user request. Every item here has caused a real incident somewhere.

01

Enable security

Set xpack.security.enabled=true, generate TLS certificates with elasticsearch-certutil, configure TLS on both HTTP (:9200) and transport (:9300) layers. Set ELASTIC_PASSWORD via Secrets Manager, not an env var in the compose file.

02

Go from 2 nodes to 3 + 3

Two master-eligible nodes can split-brain. Minimum production topology: 3 dedicated master nodes (small instances, no data) + at least 3 dedicated data nodes. The masters do only master-election work; data nodes do only data work.

03

Put a load balancer in front

No direct 9200 publishing. Use an internal ALB with HTTPS, health checks against /_cluster/health, and stickiness disabled (stateless API calls). Give clients one hostname; let the LB handle node selection.

04

Scheduled snapshots to durable storage

Register S3 as a snapshot repository. Enable SLM (Snapshot Lifecycle Management) policy: daily at 01:00 UTC, 14-day retention. Docker volumes aren’t a backup; a disk failure = permanent loss.

05

Test a restore, quarterly

An untested backup isn’t a backup; it’s a claim. Spin up a throwaway cluster, restore the snapshot, count documents. Automate it in CI if you can.

06

JVM heap at 50% of RAM, cap 31 GB

Above ~32 GB the JVM loses compressed object pointers, every reference doubles in size, you get less usable heap. Leave the other 50% of RAM for the filesystem cache — Lucene depends on it heavily.

07

Alert on the right thresholds

Cluster status ≠ green for 5 min; disk usage > 75% (watermarks kick in at 85/90/95%); JVM heap > 85% sustained; thread pool rejections > 0 for any pool. Alert on the leading indicators, not the failure.

08

Use ILM / ISM for time-series indices

Anything that grows forever (logs, events, orders by day) should roll over on a size/age limit and delete old data automatically. Without this, your cluster dies of disk exhaustion eventually — never from slowly, always at 2 AM.

09

API container: non-root, pinned digest, healthcheck

Add USER 1000 in the Dockerfile. Pin to a @sha256:... digest, not a tag. Add HEALTHCHECK in the Dockerfile. Remove CORSMiddleware(allow_origins=["*"]) — list exact origins.

10

Replace the watchdog with CloudWatch Alarms

The Go watchdog is pedagogy. In production use your cloud’s native monitoring — CloudWatch on AWS, Azure Monitor on Azure — with PagerDuty/Opsgenie routing. It has retries, escalation, on-call rotation, and audit logs that you’d spend a year building from scratch.

One-liner summary

This lab teaches the shape of an Elasticsearch cluster: discovery, shards, replication, client failover, index mappings, and monitoring. Keep the shape, swap every implementation detail for its managed-cloud equivalent, and you have a production system.
