Here’s a hands-on intro to Elasticsearch that starts with loading data, then explains indexes, mappings, shards, and why those choices matter for search quality and performance. ## 1) What Elasticsearch is doing for you Elasticsearch stores JSON documents in an **index**. An index has three core parts: **documents**, **mappings**, and **settings**. Mappings define how fields are interpreted, such as text, keyword, date, or numeric. Settings include operational choices like shard and replica counts. Search quality depends heavily on those choices because Elasticsearch builds Lucene structures from them when it indexes your data. ([Elastic][1]) A good mental model is this: **load JSON documents → Elasticsearch analyzes fields → data is written to primary shards → replicas copy that data → searches fan out across shards and merge results back together**. Writes go to a primary shard first, then replicas are updated. Searches run across shard copies, which is why shard and replica design affects both indexing and query performance. ([Elastic][2]) ## 2) The first things to learn before loading data Before you index anything, decide three things: **First, what is a document?** A document should represent one searchable thing: a product, invoice, log event, article, or customer record. **Second, what fields need exact matching versus full-text search?** For example: * `title` or `description` usually need full-text search * `status`, `category`, `email`, and IDs usually need exact match/filtering **Third, how much data and traffic do you expect?** That drives shard and replica decisions. Elasticsearch physically stores an index as shards. Primary shards hold the source data; replica shards copy primaries for fault tolerance and more read capacity. ([Elastic][1]) ## 3) Step-by-step: create your first index Create an index explicitly instead of relying on auto-created mappings. That gives you predictable search behavior and avoids bad type guesses. The create index API lets you define settings, mappings, and aliases up front. ([Elastic][3]) Example: ```bash curl -X PUT "http://localhost:9200/books" \ -H "Content-Type: application/json" \ -d '{ "settings": { "number_of_shards": 1, "number_of_replicas": 1 }, "mappings": { "properties": { "title": { "type": "text" }, "author": { "type": "keyword" }, "published": { "type": "date" }, "price": { "type": "float" }, "category": { "type": "keyword" }, "description": { "type": "text" } } } }' ``` Why this matters: * `text` fields are analyzed for full-text search * `keyword` fields are not analyzed, so they work well for filters, sorting, and aggregations * numeric and date fields support range queries efficiently Those field choices directly shape how users can search later. ([Elastic][3]) ## 4) Load a few documents Now index a few JSON documents. ```bash curl -X POST "http://localhost:9200/books/_doc/1" \ -H "Content-Type: application/json" \ -d '{ "title": "Learning Elasticsearch", "author": "Jane Smith", "published": "2025-01-15", "price": 39.99, "category": "search", "description": "A practical guide to indexing and searching documents." }' curl -X POST "http://localhost:9200/books/_doc/2" \ -H "Content-Type: application/json" \ -d '{ "title": "Distributed Search Systems", "author": "John Carter", "published": "2024-11-02", "price": 49.50, "category": "architecture", "description": "Explains shards, replicas, and query execution." }' ``` When you index a document, Elasticsearch routes it to a shard, usually based on the document ID. The write goes to the primary shard first, then the replicas. That write path is why shard count and routing affect indexing scale. ([Elastic][2]) ## 5) Load data the right way: use Bulk for real ingestion For anything beyond a few test records, use the **Bulk API**. Elastic’s docs recommend it because batching reduces per-request overhead and significantly improves indexing speed. ([Elastic][4]) Example: ```bash curl -X POST "http://localhost:9200/books/_bulk" \ -H "Content-Type: application/x-ndjson" \ -d ' { "index": { "_id": "3" } } { "title": "Lucene in Practice", "author": "A. Brown", "published": "2023-06-10", "price": 44.00, "category": "search", "description": "Covers indexing internals and relevance." } { "index": { "_id": "4" } } { "title": "Modern Data Platforms", "author": "R. Adams", "published": "2025-03-05", "price": 55.00, "category": "data", "description": "How search engines fit modern analytics platforms." } ' ``` Best practice: * use Bulk for large loads * batch in reasonable sizes * monitor failures per item, not just HTTP 200 * avoid one-document-per-request ingestion when throughput matters ([Elastic][4]) ## 6) Search the data Now run a simple full-text search: ```bash curl -X GET "http://localhost:9200/books/_search" \ -H "Content-Type: application/json" \ -d '{ "query": { "match": { "description": "indexing search" } } }' ``` A `match` query works well on `text` fields because Elasticsearch analyzes both the stored text and the query text. That is the connection between mappings and search behavior: if you mapped a field incorrectly, searches may feel “wrong” even though Elasticsearch is technically working. ([Elastic][1]) Now try an exact filter: ```bash curl -X GET "http://localhost:9200/books/_search" \ -H "Content-Type: application/json" \ -d '{ "query": { "term": { "category": "search" } } }' ``` This works because `category` is a `keyword` field. If it had been `text`, Elasticsearch would analyze it, which is usually not what you want for exact filters. ([Elastic][1]) ## 7) Shards and replicas, simply explained A **primary shard** is one partition of your index. A **replica shard** is a copy of a primary. Each document belongs to one primary shard, and replicas provide redundancy and extra search capacity. Replicas improve read scalability because searches can run against shard copies, and they improve resilience because data still exists if a node fails. ([Elastic][5]) The practical effect is: * **more primary shards** can help distribute indexing and storage * **too many primary shards** create overhead and can hurt performance * **more replicas** improve fault tolerance and read throughput * **too many replicas** consume more disk and write resources because every write must be copied ([Elastic][5]) ## 8) Good beginner shard advice For a small app or local learning project, start with: ```json { "number_of_shards": 1, "number_of_replicas": 1 } ``` That is simple and usually enough to learn with. Don’t start by over-sharding. Small clusters often run worse with too many shards because every shard has overhead. Elastic’s guidance around index fundamentals and shard architecture supports choosing shard counts deliberately rather than “more is better.” ([Elastic][1]) A simple rule for beginners: * one node in local dev: `1 shard, 0 replicas` is often practical * two or more nodes in non-prod/prod: start small, then scale based on data size, throughput, and query load Replica count is dynamic, but primary shard count is effectively a design-time decision for an index, so think harder about primaries up front. ([Elastic][6]) ## 9) Why shard count affects searching When you run a search, Elasticsearch fans the query out to relevant shards, then merges results. That means every shard adds some coordination cost. Too many small shards can make searches slower because there are more shard-level operations to manage. Too few shards can bottleneck large datasets or indexing volume. The goal is balance. ([Elastic][7]) So shard design is not just a storage concern. It directly affects: * indexing throughput * recovery time * cluster memory/metadata overhead * search latency * scaling behavior ([Elastic][5]) ## 10) Mapping best practices that improve search A lot of search problems are really mapping problems. Good defaults: * use `text` for human language content you want to search * use `keyword` for exact values, filters, joins by ID, sorting, and aggregations * use `date`, `integer`, `float`, and `boolean` explicitly * avoid letting dynamic mapping guess important field types in production unless you truly trust the input shape ([Elastic][1]) Example of a common pattern: ```json "product_name": { "type": "text", "fields": { "raw": { "type": "keyword" } } } ``` Why this pattern is useful: * `product_name` supports full-text search * `product_name.raw` supports exact filters and sorting That is often one of the best beginner design choices because it gives you both search and exact-match behavior from the same logical field. This aligns with Elastic’s mapping fundamentals and field-type guidance. ([Elastic][1]) ## 11) Ingestion best practices When loading data, think about search at ingestion time, not after. Elastic’s ingestion guidance emphasizes preprocessing, enrichment, and choosing the right field types because these have an immediate impact on retrieval quality and performance. ([Elastic][8]) Best practices: * clean and normalize data before indexing * store dates in consistent formats * separate searchable text from exact lookup fields * use Bulk API for throughput * validate documents before indexing * avoid unnecessary fields because every field adds index/storage cost * choose IDs carefully, especially if updates/upserts are common ([Elastic][4]) ## 12) A good first learning workflow Use this sequence when learning or building a POC: 1. Define one index for one document type. 2. Create the index with explicit mappings and simple shard settings. 3. Load 10–50 sample documents manually. 4. Run `match`, `term`, and range queries. 5. Switch to Bulk ingestion. 6. Inspect search results and relevance. 7. Adjust mappings before scaling data volume. That workflow works because Elasticsearch’s behavior becomes much easier to understand when you see how mapping decisions change search results. It is better than loading a huge dataset first and debugging later. ([Elastic][1]) ## 13) What beginners usually get wrong Common mistakes: * treating every string field as `text` * using dynamic mapping for everything * creating too many shards * loading large datasets one document at a time * not separating exact-match fields from full-text fields * changing design after millions of documents are already indexed These mistakes usually show up later as bad relevance, slow searches, slow indexing, or operational overhead. Elastic’s docs on index fundamentals, index settings, and bulk ingestion all point toward being explicit early. ([Elastic][1]) ## 14) A compact example you can remember Think of an online store: * `name`: `text` * `brand`: `keyword` * `sku`: `keyword` * `price`: `float` * `created_at`: `date` * `description`: `text` * `in_stock`: `boolean` Why: * shoppers search words in `name` and `description` * your app filters exactly by `brand`, `sku`, and `in_stock` * price and date need range queries That is the core “why” behind Elasticsearch design: **search fields and filter fields are not the same thing**, so they should not always be mapped the same way. ([Elastic][1]) ## 15) Where to go next After this intro, the next topics to learn are: * analyzers and tokenization * aliases and rollover/index lifecycle * relevance tuning * aggregations * update and delete patterns * scaling shard strategy over time Those build on the same foundation you just learned: document shape, mappings, settings, and shard layout.
